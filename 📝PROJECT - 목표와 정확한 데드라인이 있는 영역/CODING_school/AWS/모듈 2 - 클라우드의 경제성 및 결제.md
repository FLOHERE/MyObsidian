# 섹션 1 : 요금 기본 사항
## AWS 요금제 모델
1. 컴퓨팅 : 시간 / 초당 청구
	- 인스턴스 유형에 따라 다름
2. 스토리지 : GB 당 청구
3. 데이터 전송
	- 아웃바운드 요금은 집계하여 청구
	- 인바운드 요금은 무료
	- GB당 청구

>[!note] 인바운드와 아웃바운드
>인바운드(무료 서비스) : 외부에서 내부로 들어오는 트래픽(사용자가 간단한 이미지를 업로드 하는 등)
>아웃바운드 (유료 서비스) : 내부에서 외부로 나가는 트래픽(사용자에게 파일을 보내주는 기능 등)

## AWS의 요금 지불 방식
- 매월 말 사용량에 따라 지불 가능
- 언제든지 제품 사용을 시작하거나 중지할 수 있음
- 장기 계약 없음

1. 사용한 만큼 지불
2. 예약시 더 적은 비용 지불
3. 사용량이 증가하면 비용 절감액도 증가
4. AWS성장에 따라 더 많은 비용 절감

### 1. 사용한 만큼 지불
- 대규모 선불 지출 없이 사용한 서비스에 대해서만 비용을 지불 
- 온프레미스 : 서버에 올릴 인원수를 예상 , 거기에 맞게 선불
- AWS : 사용한 만큼만 지출
	- 사용량 많으면 많이 지불, 사용량 적으면 적게 지불
	- 모든 AWS 서비스는 온디맨드로 제공

>[!note] 온디맨드
>미리 예약 X, 쓰는 순간 바로 생성, 쓴 만큼만 비용 지불

>[!note] 인스턴스란?
> 클라우드에서 작동되는 가상의 서버를 말함

### 2. 예약시 더 적은 비용 지불(RI : Reserved Instance)
- 예약 인스턴스(RI)에 투자
- 최대 75% 비용 절감
- 옵션
	1. AURI : 전체 선불 지급 예약 인스턴스
	2. PURI : 부분 선불 지급 예약 인스턴스
	3. NURI : 더 작은 폭의 할인

#### 1. AURI : 전체 선결제 예약 인스턴스
- 절감액 최대화
- 전체를 선결제 금액으로 지불 -> 가장 큰 할인

#### 2. PURI : 부분 선불 지급 예약 인스턴스
- 할인률은 낮은 대신, 선불 지급액이 낮다.

#### 3. NURI : 더 작은 폭의 할인
- 선결제는 없지만, 할인률이 낮다. 
- 이로 인해 자금 확보 가능
- 다른 프로젝트에 활용 가능 

### 3. 사용량이 많을수록 비용 절감
- 볼륨 기반 할인 실현
	- 사용량 증가 -> 비용 절감 증가
	- Amazon Simple Storage Service(Amazon S3), 
	  Amazon Elastic File System(Amazon EFS)
	  -> 사용량이 늘수록 GB당 지불 비용 절감
- 데이터 수신 요금은 항상 무료
다
- 스토리지 서비스 포트폴리오는 데이터 엑세스 빈도 + 데이터 검색에 필요한 성능에 따라 비용을 줄일수 있는 옵션 제공 
	- 사용자가 만약 데이터 검색을 많이 하고 데이터에 자주 접근 한다면 추천 가능
	- AWS의 여러 스토리지 솔루션을 조합해서 필요한 기능만 사용 => 효율 + 비용 절감

### 4. AWS성장에 따라 더 많은 비용 절감
- 비즈니스 운영 비용을 낮춤
- 고객에게 되돌려줌

### 5. 사용자 지정 요금제
- 사용자 지정 요금제로 다양한 요구사항 충족

## AWS 프리티어 
- 무료 등급 (프리티어)을 제공
- Amazon Elastic Compute Cloud)Amazone EC2) T2마이크로 인스턴스 1년간 무료 시행 + Amazon S3 , Amazon Elastic Block Store(Amazone EBS), Elastic Load Balancing, AWS 데이터 전송 및 기타 AWS서비스에 대한 프리티어 사용 가능

## 무료 서비스 
1. Amazon Virtual Private Cloud(Amazon VPC) : AWS 클라우드에 논리적으로 격리된 공간을 프로비저닝(즉시 사용), 거기에 고객이 정의한 가상 네트워크 사용 -> AWS리소스 시작
2. AWS Identity and Access Management(AWS IAM) : 서비스, 리소스에 사용자 접근 제어 
3. Auto Scaling(통합 결제) : 여러 AWS 계정 또는 여러 Amazon Internet Services Private LImited(AISPL) 계정에 대한 결제 통합(AWS Organizations - AISPL)
	- 여러 계정에 대해 하나의 청구서
	- 각 계정의 요금을 쉽게 추적
	- 결합된 사용량에서 볼륨 요금 할인의 결과로 요금을 낮춤
	- 모든 계정 통합 -> 등급화 혜택
4. AWS Elastic Beanstalk : 애플리케이션 신속 배포, 관리
5. AWS Cloud Formation :  개발자 및 시스템 관리자가 관련 AWS리소스 모음을 쉽게 생성, 순서에 따라 예측 가능하게 프로비저닝 하도록 도움
6. 자동 크기 조정 : 사용자가 정의한 조건에 따라 리소스 자동 추가, 제거
	- 요청 급등 시 리소스 원할하게 늘어남, 성능 유지 
	- 요청 줄어들시 자동 감소 -> 인스턴스 개수 줄어듦 -> 비용 최소화
7. AWS OpsWokrs : 모든 형태와 규모의 애플리케이션 간편 배포 및 운영

>[!note] 프로비저닝이란?
>즉시 가상 서버나 데이터 베이스를 띄우는것.
>서버나 서비스에 필요한 자원을 준비, 설치 , 설정함.
>=> 실제로 바로 사용 가능한 상태를 말함


# 섹션 2 : 총 소유 비용

## 온프레미스 VS 클라우드
- 온프레미스
	- 회사(로컬)
	- 컴퓨터에 설치
	- 비용 높음
	- 확장시에도 돈 더 듦
	- 축소해도 돈은 그대로
- 클라우드 
	- 선불X
	- 사용한 만큼 지불
	- 출시기간 단축 = 민첩성 높아짐
	- 확장 , 축소 용이, 셀프 서비스 인프라
- 두 옵션의 차이 : 배포 방법

## 총 소유 비용(TCO)
- 시스템의 직/간접 비용 재정적 추정
- 쓰는 이유 
	- 온프레미스와 AWS의 비용 비교를 위함
	- 클라우드 이전을 위해 예산 책정 -> 비즈니스 사례 만들기 위함
- 워크로드 : 특정 애플리케이션이나 서비스 묶음

## TCO 고려사항
1. 서버비용
	1. 온프레미스 - 하드웨어 : 서버, 랙, 섀시의 배전 장치, 스위치 등,,
	2. AWS - 소프트웨어 : 운영체제, 가상화 라이센스 및 유지 관리
2. 스토리지 비용
	1. 온프레미스 -하드웨어 : 스토리지 디스크, SAN, 파이버 채널 스위치 등
	2. AWS - 스토리지 관리 비용
3. 네트워크 비용
	1. 온프레미스 - 네트워크 하드웨어 : LAN, 로드 밸런서 대역폭 비용
	2. AWS - 네트워크 관리 비용
4. IT 인건비 : 공통 - 서버 관리 비용

## AWS 요금 계산기
- 월벌 비용 추정
- 비용 줄이기 기회 파악
- 솔루션 구축 전 모델링
- 예상 비용에 대한 가격대와 계산 검토
- 요구에 맞는 사용 가능한 인스턴스 유형 및 계약 조건 ㅏㅈ기
- 예상 비용 이름 지정 및 서비스 그룹 생성 및 이름 지정

### 예상 비용 확인
- 총 예상 비용 = 선불 지급액(AWS스택 설정 시 금액) + 월별 예상 비용(스택 실행동안 매달 지출액)

### 추가 이점 고려
- 금전적 이점
	- 컴퓨팅 스토리지, 네트워킹 보안에 대한 지출 감소
	- 하드웨어 및 소프트웨어 구매 감소(자본 지출)
	- 운영 비용, 백업 및 재해 복구 비용 절감
	- 운영 인력 감소
- 비금전적 이점
	- 동일 클라우도 서비스 사용 -> 솔루션 정의 , 재정의 할 수 있는 서비스 및 애플리케이션 재사용
	- 개발자 생산성 향상
	- 고객 만족소 개선
	- 새로운 기회에 신속하게 대응 할 수 있는 민첩한 비즈니스 프로세스
	- 글로벌 접근성 확대

## 비즈니스 사례
- 리소스 최적화 
	- 강력한 보안 규정 준수
	- 재해 복구 개선
	- 컴퓨팅 용량 증가
- 제품 출시 속도
	- 하루만에 새로운 비즈니스 프로비저닝
	- ㅂ단 몇분만에 서비스 제공
- 운영 효율성
	- 지속적인 비용 최적화 및 절감

## 보안
1. AWS Identity and Access Management (AWS IAM) 사용 -> 액세스 제어
	1. IAM 정책 사용시 사용자, 그룹 및 역할의 액세스 허용 or 거부 가능
2. 서비스 제어 정책 (SCP) 사용시 AWS서비스에 대한 조직 구성원 단위 내 개인 또는 그룹 계정의 액세스허용 또는 거부

## 조직 설정
1. 조직 생성
	1. 현재 AWS 계정을 기본 계정으로 사용 -> 조직 생성
	2. 하나의 AWS 계정 초대 -> 조직에 가입
	3. 다른 계정을 멤버 계정으로 생성
2. 조직 구성 단위 생성(OU)
	1. 새 조직에 2개의 조직 구성 단위 생성 -> OU맴버 계정 배치
3. 서비스 제어 정책 생성
	1. 맴버 계정의 사용자 및 역할에 위임 할 수 있는 작업에 대한 제한 적용
	2. 서비스 제어 정책은 조직 제어 정책의 한 유형
4. 제한 사항 테스트
	1. 정책 테스트
	2. 각 역할에 로그인 -> 서비스 제어 정책이 계정 엑세스에 미치는 영향 확인
	3. IAM 정책 시뮬레이터 사용 -> AWS 계정의 IAM사용자, 그룹 또는 역할에 연결된 리소스 기반 정책과 IAM 테스트 -> 문제 해결
	4. 최소 권한 접근 원칙 : 보안 사고 예방(다른 사원이 실수로 설정을 건드는 등의 일 방지)

## 제한
- 이름은 유니코드
- 250자 초과 금지
- 엔티티에 대한 여러 최댓값과 최솟값이 있음
	- 이름, 계정 수 , 루트 수(1), OU 수 (1000), 제어 정책 초대장 수 (20) 등등

## 액세스
1. AWS Management Console : 조직 및 AWS 리소스를 관리하는데 사용할 수 있는 브라우저 기반 인터페이스(콘솔 사용 -> 작업 수행)
2. AWS Command Line Interface(AWS CLI) 도구 : 시스템의 명령줄에서 명령 실행 -> 태스크 수행 가능 (콘솔보다 빠를수도?)
3. AWS 소프트웨어 개발 키트(SDK) : 암호화 방식으로 서명, 오류 관리, 자동 요청 재시도 등 태스크 처리(다양한 프로그래밍 언어 지원 - python, ruby, NET, ios...)
4. HTTPS 쿼리 API : AWS Organizations , AWS 프로그래밍 방식으로 액세스 가능
	- API 사용 -> HTTPS 요청을 서비스에서 직접 실행 가능
	- 자격 증명 사용하여 요청에 디지털 방식으로 서명하는 코드 포함 필요
