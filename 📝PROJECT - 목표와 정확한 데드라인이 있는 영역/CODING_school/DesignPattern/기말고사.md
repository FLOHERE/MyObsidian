## 5장 — 싱글톤 패턴 (Singleton Pattern)

**핵심 목적:** 클래스의 인스턴스가 오직 **하나**만 존재하도록 보장

|     |                  |                                         |
| --- | ---------------- | --------------------------------------- |
|     | 키워드              | 설명                                      |
|     | `getInstance()`  | 유일한 인스턴스를 반환하는 정적 메서드                   |
|     | `uniqueInstance` | 단 하나의 인스턴스를 저장하는 정적 변수                  |
|     | `private`  생성자   | 외부에서 new로 생성 불가하게 막음                    |
|     | `enum`  싱글톤      | 가장 안전한 싱글톤 구현 방식                        |
|     | Thread-safe      | `synchronized` , double-checked locking |

**예시 클래스:**  `ChocolateBoiler`  —  `empty` ,  `boiled`  상태 관리

### 💡 핵심 개념

- **싱글톤 패턴이란?** 어떤 클래스의 인스턴스가 **오직 하나**만 존재하도록 보장하고, 그 인스턴스에 대한 **전역 접근점**을 제공하는 패턴

- **왜**  `**private**`  **생성자인가?** 외부에서  `new ClassName()` 으로 인스턴스를 직접 만드는 것을 막아, 반드시  `getInstance()` 를 통해서만 접근하도록 강제

- **멀티스레드 문제:** 두 스레드가 동시에  `getInstance()` 를 처음 호출하면 인스턴스가 2개 생길 수 있음 →  `synchronized`  또는 **double-checked locking**으로 해결

-  `**enum**`  **싱글톤:** 리플렉션(Reflection) 공격과 직렬화 문제에도 안전한, **가장 권장되는 싱글톤 구현 방법**

### 📝 예상 서술형 문제

**Q1.** 싱글톤 패턴에서  `private`  생성자를 사용하는 이유를 설명하고, 멀티스레드 환경에서 인스턴스가 2개 생성될 수 있는 문제 상황을 서술한 뒤,  `synchronized` , double-checked locking,  `enum`  세 가지 해결 방법을 비교하시오.

**Q2.**  `enum` 을 이용한 싱글톤 구현이 가장 권장되는 이유를, 리플렉션(Reflection) 공격과 직렬화(Serialization) 문제를 기존 방식과 비교하는 관점에서 설명하시오.

## 6장 — 커맨드 패턴 (Command Pattern)

**핵심 목적:** 요청(행동)을 **객체로 캡슐화** → 요청자와 수행자를 분리

|   |   |   |
|---|---|---|
||키워드|설명|
||`Command`  인터페이스|`execute()`  메서드 정의|
||`execute()`|명령 실행|
||`undo()`|명령 취소 (되돌리기)|
||Invoker|커맨드를 호출하는 객체 (ex. 리모컨)|
||Receiver|실제 동작을 수행하는 객체 (ex. Light, GarageDoor)|
||ConcreteCommand|`LightOnCommand` ,  `GarageDoorOpenCommand`  등|
||NoCommand|Null Object 패턴 — 아무것도 안 하는 기본 커맨드|
||MacroCommand|여러 커맨드를 묶어 한 번에 실행|

### 💡 핵심 개념

- **커맨드 패턴이란?** 요청(행동)을 **객체로 캡슐화**하여 요청을 보내는 쪽(Invoker)과 실제로 처리하는 쪽(Receiver)을 완전히 분리하는 패턴

- **OCP (Open-Closed Principle, 개방-폐쇄 원칙):** 새로운 커맨드를 추가할 때 기존 Invoker 코드를 **수정하지 않고 확장**만으로 기능 추가 가능 → _확장에는 열려 있고, 변경에는 닫혀 있다_

-  `**undo()**`  **원리:**  `execute()`  실행 전 상태를 저장해 두고,  `undo()`  호출 시 이전 상태로 복구 — 히스토리 스택으로 다중 undo도 구현 가능

- **Null Object 패턴:**  `NoCommand` 처럼 **아무 동작도 하지 않는 기본 객체**를 두어 null 체크 코드를 제거 — 코드 간결화

### 📝 예상 서술형 문제

**Q1.** 커맨드 패턴의 세 가지 구성 요소인 Invoker, Receiver, ConcreteCommand의 역할을 각각 설명하고, 이 패턴이 OCP(개방-폐쇄 원칙)를 어떻게 만족하는지 리모컨 예시를 들어 서술하시오.

**Q2.** 커맨드 패턴에서  `undo()`  기능이 동작하는 원리를 설명하고,  `NoCommand` (Null Object 패턴)를 사용하는 이유와 그것이 코드 품질에 미치는 영향을 서술하시오.

## 7장 — 어댑터 패턴 & 퍼사드 패턴

### 어댑터 패턴 (Adapter Pattern)

**핵심 목적:** 호환되지 않는 인터페이스를 **변환**하여 함께 사용

|   |   |   |
|---|---|---|
||키워드|설명|
||Client|기존 인터페이스(Target)를 사용하는 쪽|
||Target|Client가 기대하는 인터페이스|
||Adaptee|호환이 안 되는 기존 클래스|
||Adapter|Adaptee를 Target으로 변환해주는 클래스|
||`request()`|Client → Adapter 호출|
||`specificRequest()`|Adapter → Adaptee 내부 호출|
||**예시**|`MallardDuck`  → Turkey 어댑터|

### 퍼사드 패턴 (Facade Pattern)

**핵심 목적:** 복잡한 서브시스템을 **단순한 인터페이스 하나**로 감춤

|   |   |   |
|---|---|---|
||키워드|설명|
||Facade 클래스|복잡한 내부를 숨기는 단일 창구|
||`watchMovie()`|여러 서브시스템을 한 번에 조작|
||`endMovie()`|종료 시 복수의 객체를 정리|
||**예시**|`HomeTheaterFacade`  — 앰프, 프로젝터 등 통합 제어|
||최소 지식 원칙|Principle of Least Knowledge (데미테르 법칙)|

### 💡 핵심 개념

- **어댑터 패턴이란?** 한 클래스의 인터페이스를 클라이언트가 기대하는 **다른 인터페이스로 변환**해 주는 패턴 — 호환되지 않아 함께 쓸 수 없던 클래스를 연결

- **객체 어댑터 vs 클래스 어댑터:** 객체 어댑터는 **구성(Composition)** 사용(Java 주로 이 방식), 클래스 어댑터는 다중 상속 사용(Java 불가)

- **OCP 적용:** 기존 클라이언트 코드나 Adaptee를 수정하지 않고, **어댑터 클래스만 추가**해서 호환성 확보

- **퍼사드 패턴이란?** 복잡한 서브시스템 여러 개를 **단 하나의 단순한 인터페이스로 묶어** 제공하는 패턴 — 내부 복잡도를 감추고 사용 편의성 제공

- **최소 지식 원칙 (Principle of Least Knowledge / 데미테르 법칙):** 객체는 **꼭 필요한 대상과만 상호작용**해야 한다 — _"친구의 친구와는 대화하지 마라"_- 메서드에서 호출 허용 대상: ① 자기 자신 ② 매개변수로 받은 객체 ③ 직접 생성한 객체 ④ 인스턴스 변수로 가진 객체

### 📝 예상 서술형 문제

**Q1.** 어댑터 패턴에서 객체 어댑터(Object Adapter)와 클래스 어댑터(Class Adapter)의 차이를 구성(Composition)과 상속(Inheritance) 관점에서 비교하고, Java에서 클래스 어댑터 방식이 제한되는 이유를 설명하시오.

**Q2.** 퍼사드 패턴의 최소 지식 원칙(Principle of Least Knowledge, 데미테르 법칙)이 무엇인지 정의하고, 이 원칙을 위반하는 코드 예시와 준수하는 코드 예시를 비교하여 서술하시오.

## 8장 — 템플릿 메서드 패턴 (Template Method Pattern)

**핵심 목적:** 알고리즘의 **골격(순서)을 상위 클래스에 정의**, 세부 구현은 서브클래스에 위임

|   |   |   |
|---|---|---|
||키워드|설명|
||추상 클래스|알고리즘 골격을 정의하는 상위 클래스|
||`prepareRecipe()`|템플릿 메서드 — 단계 순서를 고정|
||추상 메서드|`brew()` ,  `addCondiments()`  — 서브클래스가 구현|
||구체 메서드|`boilWater()` ,  `pourInCup()`  — 상위 클래스에서 공통 구현|
||Hook 메서드|서브클래스가 선택적으로 오버라이드 가능한 빈 메서드|
||**예시**|`CaffeineBeverage`  →  `Coffee` ,  `Tea`  상속|
||Hollywood 원칙|"Don't call us, we'll call you" — 고수준이 저수준 호출|
||`Arrays.sort()`|Java 내장 템플릿 메서드 패턴 활용 예 (MergeSort)|

**알고리즘 순서:**

1.  `boilWater()`  → 2.  `brew()`  → 3.  `pourInCup()`  → 4.  `addCondiments()` 

### 💡 핵심 개념

- **템플릿 메서드 패턴이란?** 알고리즘의 **뼈대(실행 순서)를 상위 추상 클래스에 정의**하고, 특정 단계의 세부 구현만 서브클래스에 위임하는 패턴 — 전체 구조는 고정, 내용만 교체 가능

- **Hollywood 원칙:** _"먼저 연락하지 마세요, 우리가 연락할게요"_ — **고수준 컴포넌트(추상 클래스)가 저수준 컴포넌트(서브클래스)의 메서드를 호출**, 역방향 호출 금지

- **Hook 메서드:** 추상 클래스에 **빈 메서드**로 정의 → 서브클래스에서 선택적으로 오버라이드 가능 (ex.  `customerWantsCondiments()` ) — 필수가 아닌 선택적 확장 포인트

- **OCP 적용:** 템플릿 메서드(알고리즘 골격)는 수정하지 않고, **서브클래스 추가만으로 행동 확장** 가능

### 📝 예상 서술형 문제

**Q1.** 템플릿 메서드 패턴의 Hollywood 원칙("Don't call us, we'll call you")이 무엇을 의미하는지 설명하고, 고수준 컴포넌트(추상 클래스)와 저수준 컴포넌트(서브클래스) 간의 호출 방향이 어떻게 제어되는지 서술하시오.

**Q2.** 템플릿 메서드 패턴에서 추상 메서드(abstract method)와 Hook 메서드의 차이점을 설명하고, 각각 어떤 상황에서 사용하는 것이 적절한지 예를 들어 서술하시오.

## 9장 — 이터레이터 패턴 & 컴포지트 패턴

### 이터레이터 패턴 (Iterator Pattern)

**핵심 목적:** 컬렉션 내부 구조를 노출하지 않고 **요소를 순차적으로 접근**

|   |   |   |
|---|---|---|
||키워드|설명|
||`Iterator`  인터페이스|`hasNext()` ,  `next()` ,  `remove()`|
||`createIterator()`|메뉴 클래스에서 이터레이터 생성|
||`Iterable`  인터페이스|`java.lang.Iterable`  — for-each 지원|
||`ArrayList`  /  `HashMap`|메뉴 데이터 저장 컬렉션 타입|
||**예시**|`PancakeHouseMenu` ,  `DinerMenu`  — 각각 다른 컬렉션 사용|
||단일 책임 원칙|반복 책임을 이터레이터에 분리|

### 컴포지트 패턴 (Composite Pattern)

**핵심 목적:** 개별 객체와 복합 객체를 **동일하게 취급** (트리 구조)

|   |   |   |
|---|---|---|
||키워드|설명|
||Component|공통 인터페이스 (Leaf + Composite 모두 구현)|
||Leaf|자식 없는 단말 노드 (ex.  `MenuItem` )|
||Composite|자식을 가질 수 있는 복합 노드 (ex.  `Menu` )|
||재귀적 구조|트리 형태로 중첩 가능|

### 💡 핵심 개념

- **이터레이터 패턴이란?** 컬렉션 내부의 표현 방식(배열, ArrayList, HashMap 등)을 **외부에 노출하지 않고**, 모든 요소에 순차적으로 접근하는 방법을 제공하는 패턴

- **단일 책임 원칙 (SRP, Single Responsibility Principle):** 하나의 클래스는 **하나의 이유로만 변경**되어야 한다 → 순회(반복) 책임을 이터레이터 클래스로 분리함으로써 적용

-  `**java.util.Iterator**`  **vs**  `**Enumeration**` **:**  `Iterator` 는  `remove()`  기능 지원 + 현대적 방식,  `Enumeration` 은 레거시 → 현재는  `Iterator`  사용 권장

- **컴포지트 패턴이란?** 객체들을 **트리 구조로 구성**하여 단일 객체(Leaf)와 복합 객체(Composite)를 **동일한 인터페이스로 취급**하는 패턴 — 클라이언트는 둘을 구분할 필요가 없음

- **투명성 vs 안전성:** Component에 모든 메서드를 정의하면 투명성 ↑ / 타입 안전성 ↓, 반대도 트레이드오프 존재

### 📝 예상 서술형 문제

**Q1.** 이터레이터 패턴이 단일 책임 원칙(SRP)을 어떻게 실현하는지 설명하고,  `java.util.Iterator` 와  `Enumeration` 의 차이점을 기능 및 사용 권장 측면에서 비교 서술하시오.

**Q2.** 컴포지트 패턴에서 Component 인터페이스에 Leaf와 Composite의 모든 메서드를 정의할 때 발생하는 투명성(Transparency)과 안전성(Safety) 간의 트레이드오프를 설명하시오.

## 10장 — 상태 패턴 (State Pattern)

**핵심 목적:** 객체의 **내부 상태가 변할 때 행동도 자동으로 변경**되도록 설계

|   |   |   |
|---|---|---|
||키워드|설명|
||`State`  인터페이스|모든 상태 클래스가 구현하는 공통 인터페이스|
||Context|현재 상태를 보유하는 객체 (ex.  `GumballMachine` )|
||ConcreteState|각 상태별 행동을 구현한 클래스|
||`setState()`|상태 전환 메서드|
||`getNoQuarterState()`|상태 객체 접근 메서드|

**GumballMachine 상태 목록:**

|   |   |   |
|---|---|---|
||상태|의미|
||`NO_QUARTER`|동전 없음|
||`HAS_QUARTER`|동전 있음|
||`SOLD`|판매 중|
||`SOLD_OUT`|매진|
||`WinnerState`|당첨 상태 (보너스 껌볼)|

**주요 메서드:**

-  `insertQuarter()`  — 동전 투입

-  `ejectQuarter()`  — 동전 반환

-  `turnCrank()`  — 크랭크 돌리기

-  `dispense()`  — 껌볼 분배

-  `releaseBall()`  — 껌볼 방출

**전략 패턴 vs 상태 패턴:** 구조는 유사하나, 상태 패턴은 상태 간 **전환(transition)** 이 핵심

### 💡 핵심 개념

- **상태 패턴이란?** 객체의 **내부 상태가 바뀜에 따라 행동도 자동으로 달라지도록**, 상태 자체를 클래스로 캡슐화하는 패턴 — 마치 객체의 클래스가 바뀌는 것처럼 동작

- **if-else 지옥 탈출:** 상태마다 조건 분기(if-else if)를 쭉 나열하는 대신, 각 ConcreteState 클래스로 분산 → 유지보수성 대폭 향상

- **OCP 적용:** 새로운 상태를 추가할 때 기존 클래스를 수정하지 않고 **새 ConcreteState 클래스만 추가**하면 됨

- **전략 패턴 vs 상태 패턴:** 구조는 거의 동일 — 차이점은 **전략 패턴은 클라이언트가 알고리즘을 선택**, 상태 패턴은 **상태 객체 스스로 전환(transition)을 제어**

### 📝 예상 서술형 문제

**Q1.** 상태 패턴을 사용하지 않을 경우 발생하는 if-else 문제점을 설명하고, 상태 패턴이 이를 어떻게 해결하며 OCP(개방-폐쇄 원칙)를 어떻게 만족하는지 서술하시오.

**Q2.** 전략 패턴(Strategy Pattern)과 상태 패턴(State Pattern)은 구조적으로 유사하지만 의도(Intent)가 다르다. 두 패턴의 차이점을 알고리즘 선택 주체와 상태 전환(transition) 제어 관점에서 비교 설명하시오.

### 🔵 5장 — 싱글톤 패턴

**Q1.**  `**ChocolateBoiler**`  **클래스에 싱글톤 패턴을 적용하지 않으면 어떤 문제가 발생하는가?**

**모범 답안:**  `ChocolateBoiler` 는  `empty` ,  `boiled`  두 가지 상태로 보일러를 관리한다. 싱글톤 없이 인스턴스가 여러 개 생성되면, 한 인스턴스에서  `fill()` 로 보일러를 채우는 동안 다른 인스턴스는 이미 채워진 줄 모르고 다시  `fill()` 을 호출할 수 있다. 이는 **이중 충전, 끓지 않은 상태에서 분출** 등 물리적 오동작으로 이어진다. 따라서 보일러 제어처럼 **전역적으로 유일해야 하는 자원**에는 반드시 싱글톤 패턴이 필요하다.

**Q2.**  `**synchronized**`  **방식과**  `**double-checked locking**`  **방식의 차이점을 코드 흐름 관점에서 비교하시오.**

**모범 답안:**

|   |   |   |   |
|---|---|---|---|
||방식|동작|단점|
||`synchronized getInstance()`|메서드 전체를 동기화 → 매 호출마다 lock 획득|인스턴스 생성 후에도 불필요한 성능 저하|

double-checked locking은  `volatile`  키워드를  `uniqueInstance` 에 반드시 붙여야 메모리 가시성이 보장된다.

**모범 답안:** 이터레이터가 없으면  `Waitress`  클래스가 두 메뉴를 출력하기 위해:

**모범 답안:**
