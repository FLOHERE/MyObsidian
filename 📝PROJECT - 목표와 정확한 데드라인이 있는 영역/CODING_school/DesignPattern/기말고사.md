# 05. 싱글톤과 팩토리 패턴


## 1. Factory 패턴 개요

- **핵심 목적**: 객체 생성을 **서브클래스/별도 팩토리**에 위임해서, 클라이언트가  `new` 를 직접 많이 쓰지 않게 한다.

- **왜 쓰나?**
	- 생성 로직과 사용 로직을 분리하기 위해 
	- 구체 클래스에 대한 의존도를 낮추기 위해 
	- 객체 생성 방식이 바뀌어도 클라이언트 코드를 덜 수정하기 위해

### 시험에 잘 나오는 포인트

- **Factory Method**: 어떤 객체를 만들지는 **서브클래스가 결정**한다.

- **Abstract Factory**: **서로 연관된 객체들의 가족**을 생성하는 인터페이스를 제공한다.

- 둘 다 공통점은 **생성 부분을 캡슐화**한다는 점이다.

### 자주 헷갈리는 점

- **Factory Method**는 보통 **하나의 제품 생성 흐름**에 초점이 있고,

- **Abstract Factory**는 **제품군 전체**를 다룬다.

- 즉, Factory Method는 “무엇을 만들지”, Abstract Factory는 “어떤 조합의 제품군을 만들지”에 더 가깝다.

## 2. Factory Method 패턴

- **정의**: 객체 생성을 위한 인터페이스를 정의하고, 실제 생성은 **하위 클래스가 결정**하게 하는 패턴.

- **구성 요소**
	- **Creator**: 객체 생성을 요청하는 쪽 
	- **ConcreteCreator**: 실제 객체를 만드는 쪽 
	- **Product**: 생성될 객체의 공통 인터페이스 
	- **ConcreteProduct**: 실제 제품

### 장점

- 클라이언트가 구체 클래스에 직접 의존하지 않음

- 생성 코드가 한 곳에 모여 관리가 쉬움

- 새로운 제품 추가 시 확장에 유리함

### 단점

- 클래스 수가 늘어날 수 있음

- 구조가 복잡해질 수 있음

### 시험 포인트

- **new를 직접 쓰는 부분을 팩토리로 감싼다**는 개념

- **DIP(의존성 역전 원칙)** 관점에서 자주 연결해서 출제됨

## 3. Abstract Factory 패턴

- **정의**: 관련 있거나 의존적인 객체들의 **집합(family)** 을 생성하는 인터페이스를 제공한다.

- **핵심**: 서로 맞물리는 제품들을 **일관성 있게** 만들 수 있다.

### 예시 느낌

- 버튼, 체크박스, 텍스트박스 같은 UI 위젯을- Windows 스타일 세트 - Mac 스타일 세트 로 한 번에 생성하는 방식

### 장점

- 제품군 간 **일관성** 보장

- 구체 클래스에 대한 의존도 감소

- 제품군을 통째로 바꾸기 쉬움

### 단점

- 새로운 **제품 종류** 추가가 어려움

- 인터페이스/클래스가 복잡해질 수 있음

### 시험 포인트

- **Factory Method는 제품 생성 책임의 위임**, **Abstract Factory는 제품군 생성**

- “제품 추가 vs 제품군 추가” 차이를 묻는 문제가 자주 나옴

## ==4. Singleton 패턴==

- **정의**: 클래스의 인스턴스를 **오직 하나만** 만들고, 그 인스턴스에 **전역 접근점**을 제공하는 패턴.

- 책의 예시로는 **ChocolateBoiler** 같은 객체가 대표적이다.

### 구현 핵심

- **private 생성자**로 외부  `new` 를 막음

- **static 필드**로 유일 인스턴스 보관

- **getInstance()**로 접근

### 대표 구현 방식

1. **지연 초기화(lazy initialization)**- 필요할 때 처음 생성 - 간단하지만 멀티스레드에서 문제 가능

2. **synchronized getInstance()**- 스레드 안전하지만 성능 저하 가능

3. **Double-Checked Locking(DCL)**- 먼저 null 체크 후, 동기화 블록 안에서 다시 체크 - 보통 `volatile`과 함께 사용

4. **static initializer**- 클래스 로딩 시 바로 생성 - 단순하지만 늦게 만들 수 없음

5. **enum Singleton**- 가장 안전한 방법으로 자주 언급됨 - 직렬화/리플렉션 문제에 강함

### 장점

- 객체를 하나로 통제 가능

- 전역 접근이 쉬움

- 공유 자원을 관리하기 좋음

### 단점

- 전역 상태처럼 쓰이기 쉬워 테스트가 어려울 수 있음

- 남용하면 결합도가 높아질 수 있음

- 멀티스레드 환경에서는 구현에 주의 필요

### 시험 포인트

- **private constructor + static instance + getInstance()** 조합을 정확히 기억

- **멀티스레드 안전성**이 핵심

- **DCL에서**  `**volatile**` **이 왜 필요한지** 자주 물어봄

- **enum Singleton이 안전한 이유**를 묻는 문제 가능

## 5. ChocolateBoiler 예제로 보는 Singleton

- 상태 예시:- `empty` - `boiled`

- 핵심 동작:- `fill()` → 비어 있을 때만 채움 - `boil()` → 비어 있지 않고, 아직 끓지 않았을 때만 끓임 - `drain()` → 비어 있지 않고, 끓었을 때만 비움

### 시험에 나올 포인트

- **생성자 접근 제한**으로 인스턴스 중복 생성 방지

- 상태 변화 조건을 통해 **객체의 일관성 유지**

- Singleton은 단순히 “하나만 만든다”가 아니라, **상태 관리의 통제**까지 포함한다

## 6. 한 번에 정리하는 비교

|   |   |   |   |
|---|---|---|---|
||패턴|핵심 목적|시험 포인트|
||Factory Method|생성 책임을 하위 클래스에 위임|어떤 객체를 만들지 서브클래스가 결정|
||Abstract Factory|관련 객체군을 일괄 생성|제품군의 일관성, 제품 추가와의 차이|
||Singleton|인스턴스를 하나만 유지|private constructor, getInstance(), thread safety|

## 7. 최종 암기 포인트

- **Factory Method**: “생성은 서브클래스가 결정”

- **Abstract Factory**: “관련된 객체군을 한 번에 생성”

- **Singleton**: “인스턴스는 하나, 접근은 전역”

- **DCL + volatile**: 멀티스레드 안전성 때문에 중요

- **enum Singleton**: 안전성과 간결성 때문에 자주 언급됨

# 06. 커맨드 패턴

## 1. 커맨드 패턴의 구조

커맨드 패턴은 다음과 같은 구성 요소로 이루어집니다:

| 구성 요소               | 역할                                      | 실생활 비유                   |
| ------------------- | --------------------------------------- | ------------------------ |
| **Command (인터페이스)** | `execute()`  메서드를 선언하는 인터페이스            | "주문서" 양식                 |
| **ConcreteCommand** | 실제 동작을  `execute()` 에 구현. Receiver를 호출  | "햄버거 주문서", "쉐이크 주문서"     |
| **Client**          | ConcreteCommand 객체를 생성하고 Invoker에 전달    | "손님"                     |
| **Invoker**         | Command 객체를 보관하고 필요할 때  `execute()`  호출 | "웨이트리스" (주문서를 받아 주방에 전달) |
| **Receiver**        | 실제 작업을 수행하는 객체                          | "주방장" (실제 요리를 함)         |
